Using Git in Atom

When you are working on a project with Git, you have two main places where your code is kept: a local repository, which is on your personal computer, and a remote repository, on a code-hosting site such as GitHub.

Why do we need two locations? Your local repository is where the work happens– this is where you do the actual coding and testing. On the other hand, you can think of your remote respository as the current “best version” of your code: it doesn’t have to be perfect, but it’s used as a jumping-off point for any future work you and others may do. Having a remote repository also means that others can look at your code, and that your code is safe if something were to happen to your personal computer.

Lesson 0: Starting the project

There are 2 ways to start working on a Git project on your personal computer: initializing and cloning. Each one is used in different situations.

  • Cloning (Recommended) Cloning is for when you or someone else already has a project in a remote repository on GitHub, and you want to make a copy of it in a local repository so you can make some changes to the project.
    1. Go to GitHub and find the project you want to clone. If you’re starting a new project, create the project now.
    2. From the main project page, click “Clone or download”.
    3. Click the copy to clipboard button to copy the .git URL.
    4. In Atom, open the Command Palette with Command + Shift + P and search for “GitHub: Clone”.
    5. Paste the .git URL from Step 3 into the “Clone from” field, and select a new folder to create the local repository in.
    6. Click “Clone”. You should now have a local copy of the project in the folder you just named!
  • Initializing Initializing is for a brand new project, for which neither you nor anyone else has written any code. For these, you start by initializing your local repository.
    1. If you haven’t already done so, create a repository in GitHub.
    2. Follow the instructions in “Cloning” to find the .git URL for your GitHub repository (Steps 2 and 3 only). Keep this URL handy; we’ll come back to it later.
    3. Open the Git panel in Atom.
    4. Click “Create Repository”.
    5. Important: Carefully choose the folder in which you want to create the repository. If you want to start MyNewProject in your CSSI folder, you should choose the path “Users/demouser/Desktop/cssi/MyNewProject” (exact folder names may vary). If you initialize in the wrong folder, you’ll end up either pushing too much or too little to GitHub.
    6. Click “Init+”.
    7. Go to your Terminal and cd into the project directory of your local repository (e.g., /Users/demouser/Desktop/cssi/MyNewProject).
    8. Copy the .git URL from step 1, and run this command: git remote add origin <your-git-url-goes-here> (leave off the <> when you substitute your .git URL).

Lesson 1: Committing changes

As you might remember, you are modifying your code in your working copy, while you are also keeping a history of your and your partners’ work in a local repository, stored offline, in your computer, and a remote repository, in our case stored in GitHub.

While there is only one remote repository, each teammate is making changes on their own working copy, and logging them in a separate local repository. As a matter of fact, if you’re working from two computers, or even two folders in the same computer, you can have multiple local repositories for yourself.

How do we keep everything synced? Fortunately, git is a very powerful tool.

Commiting local changes

In a normal workflow, you will be making local commits (on your computer) very often during the day so that you can keep track of all changes you make, and so that you can be sure that you can always go back to a previous working status, in case some of your changes break your code.

This step has two parts: “staging” the files you want to commit (also called “adding”), and then actually “committing” your change.

  1. In the Git button in Atom, you will see the number of files you have changed in your current repository. Click on this button to open the Git panel.
  2. Stage the files you want to go into your change. You can stage individual files by right clicking on them and selecting “Stage”, or you can stage all files by clicking “Stage All” in the upper right-hand corner.
  3. Once you made all the changes you want to and have staged all the files you want to include, we’re almost ready to commit! The last thing you have to do is write a commit message. The commit message should be a short description of what you’re changing in the commit; i.e., “Add a sign in link to the Home page”, or “Fix a bug in the student handler where changes to students weren’t saving.” Some examples of bad commit messages: “Bug fix”, “2nd try”, “stuff”, etc.
  4. Once you’ve written a commit message, you can click “Commit”!

A note on commits

One commit should encapsulate exactly one self-contained change. They should generally be pretty granular: for example, a commit could be a single bug fix (even as small as fixing a typo!), or adding a single function, or even as large as a small feature. The smaller and more self-contained they are, the easier it will be to work on your final project with your team.

A good rule of thumb: if you put an “and” in your commit message (i.e., “Clean up CSS on home page, and add a click listener”), your commit is already too big!

Pushing to remote repository

If you cloned your repository from GitHub, or have pushed to your local repository to GitHub before, pushing is very simple: all you have to do is press “Push”! You will have to enter your GitHub username and password.Note: The number next “Push” is the number of commits you are pushing.

Pulling from a remote repository

There are many reason why the remote repository might have more changes than the ones you have on your local repository:

  • You might be working in a team, and one of your team members has recently pushed a new change.
  • You might be working on two computers, and the one you’re on now does not have the most up-to-date version of your code.

In both cases, it is quite simple to retrieve the updated status from the remote repository:

  1. Find the “Fetch” button in lower right hand corner of Atom.
  2. Right click and select “Pull”.

Pull is an important step to make sure you have the most up to date version of the repo on your local computer. This is most important when multiple people are working on and committing changes to the same repo.

What we’ve learned

Git and GitHub offer a service that is invaluable for a programmer. Even when we are not collaborating with more people, it is very useful to have a complete history of our changes, so that we can always go back to a previous state if necessary.
