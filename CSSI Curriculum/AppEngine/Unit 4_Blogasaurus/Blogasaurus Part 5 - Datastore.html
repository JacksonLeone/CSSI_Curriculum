Blogasaurus Part 5: Datastore

In this section, we will start storing the blog posts our users submit in a database, so our app will remember them and be able to display them later.

See a demo for what this step could look like. You can also view the code for the demo, but try coding your own version before looking at the demo code.

If you need to restart your project for whatever reason, expand the section below for instructions.

To start fresh, you will be downloading the demo code from the last step into a new directory.

First, in the Terminal, navigate to your home folder. You do not want to be in the folder for your old Blogasaurus project.

Then, run the following command:

1

git clone -b step-4-templates --single-branch https://github.com/google/cssi-blogasaurus.git blogasaurus_v5

This will create a new folder called blogasaurus_v5 that contains the demo code from part 4 of the Blogasaurus project. From here, you can continue with the instructions for part 5.

Step 1: Create a Model

Navigate to your blogasaurus folder and create a new file called models.py. In this new Python file, write a Model called BlogPost. It should include the data you requested in the form you wrote previously:

  • The title of the blog post
  • The content of the blog post
  • The author’s name

Don’t forget to import ndb!

Step 2: Start saving posts

In main.py, you wrote a post method that retrieves data from a form and renders it into a new template. Use the Model you just wrote to first save this data in the database.

Check-in: Start your local server and test this by adding a new post. Then, open up the Datastore viewer in the App Engine Admin console. You should see a new Entity! Check to make sure the correct values got saved in the correct fields.

Step 3: Query all posts

In this and the following step, we’re going to update the page our app redirects to after the user saves a post to show not just the one post they just created, but all of the previously created posts.

In your post method, write a datastore query to get all saved posts. This will return a list. Add the new BlogPost object you created to the front of this list.

Note: You want to do this query before you save the new post into the database. Expand below for an explanation of why we want to do it this way.

You’re saving the new BlogPost object to the database, so you would expect that if you query the database after you save it, you should get the new BlogPost object you just created. In an ideal world, that would be true; often in your local server, it is true.

However, the deployed version of Datastore is distributed, meaning it stores data across multiple locations. This is for a couple of reasons: 1) it stores a backup of your data, to prevent data loss, and 2) it expects people to access this data from many places, and so it wants to store the data in multiple places to make sure it gets to those people as quickly as possible. Because of this distibuted model, it means that Datastore is not immediately consistent, meaning the data isn’t propagated everywhere immediately.

It doesn’t take long for everything to be consistent (usually about 30 seconds), but that’s much slower than the code handler. Therefore we should pull the list before you’ve saved the object, and manually add the new object to the list. That way we’re guaranteed to have the object in the list 1 time, instead of 0 (if we assume the object is in the list when it hasn’t actually propagated yet), or 2 times (if saving moves fast, as it does in the local server, and we’ve saved it to the database, retrieved it with the rest of the list, and then added it manually ourselves).

Check-in: Use the print function to log the list of blog posts you just queried to the Terminal. Make sure it’s getting all the posts you created. Once you’ve checked that it’s working as expected, remove the print statement so you don’t clutter up your logs.

Right now in our template variables dictionary, we’re only sending the most recently created post, rather than all posts. Change the key in the dictionary to a more descriptive name (i.e., "blog_posts"), and change the value to the list we just queried.

Step 4: Update your results template

Find the template you named view_post.html in the previous step, and rename it view_all_posts.html. Don’t forget to also update the name in your post handler.

Update the template to show all the blog posts, rather than just the one. Make sure you use the variable name you just created!

Hint: You should be able to reuse most of the HTML you already wrote! See the jinja2 documentation for instructions on how to loop through your list of blog posts.

Check-in: Run your local server. Try adding some new posts. You should see all the posts you’ve created so far after you submit the form!

Stretch: Order the blog posts by the time they were created

The first thing you’ll have to do here is start saving the date and time the posts were created.

Hint 1: Google how to find the current date and time in Python (if you did the stretch activity in the Templates Blogasaurus lab, then you’ve already done this). This will give you the value you want to save to the database.

Hint 2: Look at the Datastore documentation to find the most appropriate Property type for this field.

Check-in: Once you’ve added the property to the model and started saving it, try running your local server and creating a blog post. Go to your App Engine Admin console to make sure your new BlogPost entity has the date and time saved on it.

At this point, most of your saved data won’t have the date and time field(s) stored. Use the App Engine Admin console to delete all the BlogPost entities that don’t have data on them.

Use your app to create a few more blog posts.

In order to ensure that our list of blog posts gets displayed in the order they were created (most recent first), add an order onto the query you wrote before.

Check-in: Run this a few more times. Your app should always display the blog posts in the order they were created!
